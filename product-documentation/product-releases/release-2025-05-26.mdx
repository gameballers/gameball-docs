  <Accordion title="Exclude Taxes from Reward Calculations" icon="percent">
    **Platform:** All Platforms | **Plan:** By Request | **Channel:** All

    We've introduced a new configuration option allowing you to **exclude taxes from reward calculations**, giving greater accuracy and control over how cashback or points are distributed. [Learn more about points redemption options](/product-documentation/programs/loyalty-points-redeem/launch-and-configure-your-redeem-pointing-system).

    <CardGroup cols={3}>
      <Card title="Financial Precision" icon="calculator">
        Reward customers based only on the true product value, excluding tax amounts.
      </Card>
      <Card title="Loyalty Accuracy" icon="bullseye">
        Prevent over-rewarding by basing rewards on net purchases, not tax-inflated totals.
      </Card>
      <Card title="Global Compliance" icon="globe">
        Especially valuable for regions with high or variable tax rates.
      </Card>
    </CardGroup>

    **How It Works:**

    A new toggle is available under **Earn Settings** — labeled "Exclude Taxes from Reward Calculations." It is disabled by default. When enabled, all rewards (Cashback, Points, etc.) will be calculated based on the order subtotal before taxes.

    **API Requirements:**

    To enable this functionality, you must send the following fields in order-related APIs:

    | Field | Description |
    |-------|-------------|
    | `total_tax` | Total tax amount for the order |
    | `taxes` (per lineitem) | Breakdown of tax amounts per item |

    <Warning>
      If tax data is **not provided** while the toggle is **enabled**, Gameball will default to using the full order amount (including tax).
    </Warning>
  </Accordion>

  <Accordion title="New API: Retrieve Coupon Lock Details" icon="lock">
    **Platform:** Enterprise | **Plan:** Clients Using our Couponing Engine | **Channel:** All

    We've released a new API endpoint that allows you to **fetch detailed information about locked coupons**, giving more transparency and control over coupon management.

    <CardGroup cols={3}>
      <Card title="Operational Clarity" icon="eye">
        Helps teams understand who has a locked coupon, when it was locked, and other key metadata.
      </Card>
      <Card title="Issue Resolution" icon="wrench">
        Speeds up investigation and debugging related to coupon redemption and locking behavior.
      </Card>
      <Card title="Optimized Experience" icon="gear">
        Proactively manage coupon lock durations, re-locking behavior, and customer-specific edge cases.
      </Card>
    </CardGroup>

    **API Overview:**

    - **Endpoint:** `GET /integrations/locks/{lockReference}`
    - **Purpose:** Returns detailed info for a locked coupon using the `lockReference`.

    **Key Response Data Includes:**

    | Field | Type | Description |
    |-------|------|-------------|
    | `CustomerId` | string | Associated customer |
    | `isUsed` | Boolean | Whether the coupon has been used |
    | `usedAt` | timestamp | When the coupon was used |
    | `isExpired` | Boolean | Whether the coupon has expired |
    | `ExpiredAt` | timestamp | When the coupon expired |
    | `coupons` | array | List of all locked coupons under this reference |

    [View Full API Reference](https://developer.gameball.co/rest-api/coupons#get-lock-details)
  </Accordion>
</AccordionGroup>

## Other Improvements

<AccordionGroup>
  <Accordion title="Dynamic Rewards Now Support Up to 3 Decimal Places" icon="decimal">
    Dynamic Rewards now support multipliers with up to **three decimal places**, allowing for more accurate point distribution (e.g., 1.015 instead of 1.5).

    **Example:** A $1000 spend with a 1.015 multiplier earns **1015 points**. [Learn more about rewards](/product-documentation/rewards-campaigns/faqs/understanding-rewards-types).
  </Accordion>

  <Accordion title="Improved Automation Reactivation Handling" icon="rotate">
    Previously, deactivating an automation would remove all customers from it, causing a loss of their progress when the automation was reactivated. Now, customers' progress within the campaign is preserved. When a customer deactivates and later reactivates an automation, the system saves each customer's position in the flow, allowing them to continue exactly where they left off.
  </Accordion>

  <Accordion title="Game Lead Form Enhancements: Update Only Missing Customer Profile Attributes" icon="user-pen">
    When using a game campaign with a lead form, previously, if a customer who was **already registered** submitted new data, that data was ignored. Now, Gameball uses only the newly entered data and adds it to the customer's profile, ensuring no existing data is overwritten.
  </Accordion>
</AccordionGroup>
